# Automated 424B4 Filing Pipeline

## Overview

This document describes the fully automated pipeline for monitoring, downloading, and processing 424B4 IPO filings using Claude Batch API (50% cost savings).

**Pipeline Flow:**
```
SEC RSS Feed → Download PDF → Extract Page 1 → Upload to Blob
                                                    ↓
                                            Submit to Batch API
                                                    ↓
                                            Poll for Results
                                                    ↓
                                            Store in Neon DB
```

## Architecture

### Cron Jobs (Vercel)

1. **check-new-filings** (Every hour at :00)
   - Monitors SEC RSS feed for new 424B4 filings
   - Adds new filings to database with status='pending'
   - Path: `/api/cron/check-new-filings`

2. **submit-batch** (Every 2 hours at :15)
   - Downloads PDFs from SEC
   - Extracts first page only (prospectus summary)
   - Uploads to Vercel Blob storage
   - Creates batch request for Claude API
   - Submits up to 20 filings per batch
   - Path: `/api/cron/submit-batch`

3. **poll-batches** (Every 30 min at :30 and :45)
   - Checks status of active batches
   - Processes completed batches
   - Extracts metrics and stores in database
   - Path: `/api/cron/poll-batches`

### Database Schema

#### `filings` table
```sql
CREATE TABLE filings (
  id SERIAL PRIMARY KEY,
  ticker VARCHAR(10),
  cik VARCHAR(10),
  accession_number VARCHAR(30) UNIQUE,
  filing_date DATE,
  filing_url TEXT,
  blob_url TEXT,
  status VARCHAR(20),  -- pending, processing, completed, failed
  batch_id VARCHAR(100),
  batch_custom_id VARCHAR(100),
  batch_submitted_at TIMESTAMP,
  batch_completed_at TIMESTAMP,
  retry_count INT DEFAULT 0,
  error_message TEXT,
  created_at TIMESTAMP DEFAULT NOW(),
  processed_at TIMESTAMP
);
```

#### `batch_jobs` table
```sql
CREATE TABLE batch_jobs (
  id SERIAL PRIMARY KEY,
  batch_id VARCHAR(100) UNIQUE NOT NULL,
  status VARCHAR(20) NOT NULL,
  filing_count INT NOT NULL,
  submitted_at TIMESTAMP DEFAULT NOW(),
  completed_at TIMESTAMP,
  expires_at TIMESTAMP,
  error_message TEXT,
  result_url TEXT,
  request_counts JSONB
);
```

#### `pipeline_state` table
```sql
CREATE TABLE pipeline_state (
  id SERIAL PRIMARY KEY,
  last_rss_check TIMESTAMP,
  last_successful_filing TIMESTAMP,
  total_processed INT DEFAULT 0,
  updated_at TIMESTAMP DEFAULT NOW()
);
```

## Deployment Instructions

### 1. Setup Database Schema

Run the database migration to add batch tracking:

```bash
npm run db:add-batch-tracking
```

This creates the necessary tables and indexes.

### 2. Configure Environment Variables

Add these to your Vercel project settings:

**Required:**
- `DATABASE_URL` - Neon PostgreSQL connection string
- `ANTHROPIC_API_KEY` - Claude API key
- `BLOB_READ_WRITE_TOKEN` - Vercel Blob storage token
- `SEC_USER_AGENT` - Your contact info (e.g., "YourCompany support@example.com")

**Auto-generated by Vercel:**
- `CRON_SECRET` - Automatically set for cron authentication

### 3. Deploy to Vercel

```bash
# Commit changes
git add .
git commit -m "feat: add automated 424B4 pipeline with Batch API"

# Deploy to production
vercel --prod
```

Vercel will automatically:
- Set up the 3 cron jobs from `vercel.json`
- Generate `CRON_SECRET` for authentication
- Enable the scheduled tasks

### 4. Verify Deployment

After deployment, check the cron jobs in Vercel dashboard:
1. Go to your project → Settings → Cron Jobs
2. Verify all 3 jobs are listed and enabled
3. Check recent executions and logs

### 5. Test the Pipeline

**Option A: Trigger Manually (for testing)**

You can trigger each cron job manually via API:

```bash
# Get CRON_SECRET from Vercel dashboard
CRON_SECRET="your_secret_here"

# Test RSS checker
curl -X GET "https://your-domain.vercel.app/api/cron/check-new-filings" \
  -H "Authorization: Bearer $CRON_SECRET"

# Test batch submission
curl -X GET "https://your-domain.vercel.app/api/cron/submit-batch" \
  -H "Authorization: Bearer $CRON_SECRET"

# Test batch polling
curl -X GET "https://your-domain.vercel.app/api/cron/poll-batches" \
  -H "Authorization: Bearer $CRON_SECRET"
```

**Option B: Wait for Automatic Execution**

The cron jobs will run on their schedules:
- :00 - Check for new filings
- :15 - Submit batch (every 2 hours)
- :30, :45 - Poll for results

## Cost Estimation

### Claude Batch API (50% savings vs regular API)

- **Regular Sonnet 4.5**: $3 per 1M input tokens
- **Batch API Sonnet 4.5**: $1.50 per 1M input tokens

**Per Filing:**
- Page 1 PDF: ~5K tokens (image)
- Prompt: ~1K tokens
- Response: ~2K tokens
- **Total per filing**: ~$0.012 (batch) vs $0.024 (regular)

**Monthly (100 filings):**
- Batch API: $1.20/month
- Regular API: $2.40/month
- **Savings**: $1.20/month (50%)

### Vercel Costs

- **Cron Jobs**: Free on all plans
- **Function Invocations**:
  - Hobby: 100GB-hrs/month free
  - Expected: ~5-10 GB-hrs/month (well within limits)
- **Blob Storage**:
  - $0.15/GB/month
  - 100 filings × 1 page × 500KB = ~50MB = $0.007/month

### Total Monthly Cost: ~$1.50

## Processing Timeline

### Batch API Processing Time

- **Submission to Results**: Typically 30 minutes to 2 hours
- **Maximum**: 24 hours (batch expires after 24h)
- **Most common**: 45-90 minutes

### End-to-End Timeline

For a new 424B4 filing published by SEC:

1. **Detection**: Within 1 hour (RSS check runs hourly)
2. **Download & Upload**: Within 2 hours (submit-batch runs every 2h)
3. **Batch Processing**: 30-120 minutes (Claude Batch API)
4. **Results Available**: Within 30 minutes (poll-batches runs twice/hour)

**Total**: Typically 2-4 hours from SEC publication to database

## Handling Failures

### Automatic Retries

The pipeline includes retry logic:

1. **Download Failures**: Filing stays in 'pending' status, retried on next run
2. **Batch Failures**: Status marked 'failed', retry_count incremented
3. **Extraction Errors**: Logged with error_message, can be manually reviewed

### Manual Intervention

If a filing continuously fails:

```sql
-- Check error message
SELECT ticker, status, error_message, retry_count
FROM filings
WHERE status = 'failed'
ORDER BY retry_count DESC;

-- Reset for retry
UPDATE filings
SET status = 'pending',
    error_message = NULL,
    retry_count = 0
WHERE id = {filing_id};
```

## Troubleshooting

### No New Filings Detected

**Check:**
1. SEC RSS feed accessibility: `curl https://www.sec.gov/cgi-bin/browse-edgar?action=getcurrent&type=424B4&output=atom`
2. Last RSS check time in pipeline_state table
3. Cron job execution logs in Vercel

**Fix:**
- Manually trigger: `curl .../api/cron/check-new-filings`
- Verify `SEC_USER_AGENT` is set correctly

### Batch Stuck in Processing

**Check:**
1. Batch status via Claude API dashboard
2. Expiration time (batches expire after 24h)
3. Batch job record in database

**Fix:**
```sql
-- Check batch status
SELECT * FROM batch_jobs WHERE status IN ('submitted', 'processing');

-- If expired, mark as failed
UPDATE batch_jobs
SET status = 'expired', error_message = 'Batch expired after 24h'
WHERE batch_id = 'xxx' AND expires_at < NOW();

-- Reset filings for retry
UPDATE filings
SET status = 'pending', batch_id = NULL
WHERE batch_id = 'xxx';
```

### High Failure Rate

**Common Causes:**
1. Incorrect PDF URL pattern (SEC URL structure changed)
2. Network timeouts
3. Blob storage quota exceeded
4. Claude API rate limits

**Debug:**
```sql
-- Analyze error patterns
SELECT
  error_message,
  COUNT(*) as count
FROM filings
WHERE status = 'failed'
GROUP BY error_message
ORDER BY count DESC;
```

## Advanced Configuration

### Adjust Batch Size

Edit `/api/cron/submit-batch.ts`:

```typescript
// Change from 20 to desired size (max 10,000 per batch)
const filingsForBatch = await getFilingsForBatch(50);
```

**Trade-offs:**
- Larger batches: Fewer API calls, longer processing time
- Smaller batches: Faster results, more frequent updates

### Change Processing Frequency

Edit `vercel.json`:

```json
{
  "crons": [
    {
      "path": "/api/cron/check-new-filings",
      "schedule": "*/30 * * * *"  // Every 30 min instead of hourly
    }
  ]
}
```

### Add Email Notifications

Install email service (e.g., SendGrid):

```bash
npm install @sendgrid/mail
```

Add to `/api/cron/poll-batches.ts`:

```typescript
if (failedCount > 0) {
  await sendEmail({
    to: 'admin@example.com',
    subject: `Filing Pipeline: ${failedCount} failures`,
    body: `Check dashboard for details`
  });
}
```
